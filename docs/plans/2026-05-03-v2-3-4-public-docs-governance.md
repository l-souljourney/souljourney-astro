# v2.3.4 公开文档治理与历史材料收敛

**日期：** 2026-05-03  
**版本：** `v2.3.4`  
**状态：** active  
**目标：** 将 `souljourney-blog` 的公开文档面收敛为适合 GitHub 开源访问的博客平台文档体系，统一版本口径，压缩不必要的运维细节暴露，并对历史材料做公开级别治理。

---

## 1. 为什么需要 `v2.3.4`

在 `v2.3.3` 之前，这个仓库的文档既承担了开发记录、运维审计、历史计划，也承担了开源项目的公开入口。

这带来了三类问题：

1. 版本口径漂移：
   - `README`
   - 路线图
   - `update.md`
   - 历史任务记录
   没有完全对齐到同一当前状态。
2. 公开深度过深：
   活跃文档中混入了过多运维实现、平台细节和历史验证痕迹。
3. 历史材料边界不清：
   一部分文档有历史参考价值，但不适合继续以原始工作记录形态长期公开保留。

因此 `v2.3.4` 不是功能升级版，而是一个面向开源公开面的文档治理补丁版。

---

## 2. 本版目标

`v2.3.4` 只做以下事情：

- 统一 README、docs 索引、roadmap、`update.md` 的版本口径
- 收缩活跃文档公开深度
- 明确哪些历史资料可以公开归档，哪些只能保留摘要
- 清理明显不适合长期公开暴露的运维/研究记录

---

## 3. 公开文档原则

从 `v2.3.4` 起，公开主入口文档只保留：

- 项目定位
- 当前版本状态
- 内容契约
- 架构边界
- 协作所需的中层说明

以下内容不再作为公开主入口文档的一部分：

- secrets / vars 名称
- 私有平台地址或环境导入路径
- 运维执行命令
- 平台验证证据、run id、构建审计细节
- 历史安全事件的原始处理记录

---

## 4. 治理范围

本轮治理范围包括：

- `README.md`
- `docs/**`
- `update.md`
- `.trellis/tasks/archive/**`

本轮不包括：

- `.trellis/workspace/**`
- Astro 业务功能升级
- 远端平台设置调整

---

## 5. 验收标准

`v2.3.4` 完成时，至少满足：

- [ ] `README.md`、`docs/README.md`、roadmap、`update.md` 对当前版本口径一致
- [ ] 活跃公开文档不再继续暴露 secrets / vars 名称、平台私有地址、运行证据和运维执行细节
- [ ] 历史 docs 与 Trellis archive 完成一轮公开级别分流
- [ ] 公开保留的 deploy / integration 文档降到“中层说明”
- [ ] 现有站点构建验证仍通过

---

## 6. 与 `v2.4` 的关系

`v2.3.4` 是 `v2.3.3` 之后的治理补丁版，用于先把公开文档面收紧。

`v2.4` 仍保留为后续的 Astro 升级与自动化质量保障阶段，不在本版内展开。
