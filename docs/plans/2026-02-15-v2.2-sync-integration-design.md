# Design Document: v2.2 GitHub Sync Hub Integration

**Date:** 2026-02-15
**Status:** Approved
**Version:** v2.2

## 1. Overview
The goal of this version is to integrate the `souljourney-blog` repository with the central `souljourney-code` management hub. This allows for automated synchronization of tasks and progress from this repository to a centralized Notion dashboard (via the Hub).

## 2. Architecture
This repository will act as a **Spoke**. It will emit events via GitHub Actions to the **Hub** (`l-souljourney/souljourney-code`) using the `repository_dispatch` mechanism.

## 3. Detailed Design

### 3.1 GitHub Workflow: `dispatch-to-hub.yml`
- **Path:** `.github/workflows/dispatch-to-hub.yml`
- **Events to Track:**
    - `issues`: All types (opened, edited, deleted, closed, reopened, labeled, unlabeled, assigned, unassigned, milestoned, demilestoned).
    - `pull_request`: Type `closed`.
- **Payload Structure:**
    - `event_type`: `github_sync`
    - `client_payload`:
        - `repo`: `l-souljourney/souljourney-blog`
        - `event`: Full event context provided by GitHub.

### 3.2 Security & Authentication
- Uses a Personal Access Token (PAT) stored in GitHub Secrets as `GH_SYNC_TOKEN`.
- The token must have `repo` scope or `contents: write` on the target Hub repository.

## 4. Acceptance Criteria
- [ ] Workflow triggers correctly on Issue creation.
- [ ] Workflow triggers correctly on PR closure.
- [ ] GitHub API returns a 204 status code upon dispatch.
- [ ] No regression on existing deployment workflows.

## 5. Risk Assessment
- **Token Expiry:** If the PAT expires, sync will stop. Mitigation: Use a long-lived token or fine-grained PAT with alerts.
- **Dispatch Loop:** Unlikely in this Spoke-Hub model as the Hub does not echo back the same event to this repo.
