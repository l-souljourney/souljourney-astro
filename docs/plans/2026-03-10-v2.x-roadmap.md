## 架构定位（重构后口径）

### 1. 发布系统边界

Astro 仓库只负责以下职责：

- 消费 GitHub 仓库中的 Markdown/MDX 内容
- 对双语文章进行 schema 校验、镜像配对、路由生成、SEO 输出、搜索索引生成
- 进行静态构建与部署

以下能力视为**外部已实现前置条件**，不再作为 Astro 主路线目标：

- Obsidian 触发发布
- HTTP 发布入口
- GitHub 自动提交/触发构建

### 2. 双语发布唯一标准

后续所有规划统一遵循以下强约束：

- `source_id` 是跨语言内容主键，zh/en 共用且稳定
- `slug` 是跨语言共享路由键，zh/en 保持一致
- 中文文章路由保持 `/article/{slug}`
- 英文文章路由保持 `/en/article/{slug}`
- zh/en 必须成对存在，才进入正式发布集合
- 单边内容视为 `pending_translation`，不得进入 article route、RSS、sitemap、搜索索引与公开聚合页
- 发布与下架默认按双语成对处理（both）

### 3. 当前主矛盾

在未来 1000+ 篇内容规模下，真正需要治理的不是“是否继续对接 Engine”，而是：

- 双语内容模型是否稳定
- 路由与模板是否可持续维护
- 搜索是否能按语言正确分桶
- 构建时间与产物体积是否可控
- 客户端脚本是否只在需要的页面运行

## v2.2 — 双语镜像发布治理 + Astro 核心收敛

**目标：** 把当前博客从“补丁式双语站点”收敛成“可持续双语发布平台”  
**依赖：** 无需新增外部发布系统能力，基于现有 GitHub 发布链路即可开始  
**预估工作量：** 5-7 天

### 核心范围

#### A. 双语镜像发布治理

| # | 任务 | 说明 |
|---|------|------|
| A1 | 固化双语镜像规则 | `source_id` / `slug` 成为镜像内容的唯一约束主键 |
| A2 | 定义公开发布集合 | 只有完整 zh/en 镜像对进入 article/RSS/sitemap/search |
| A3 | 定义未配对内容策略 | 单边稿件视为 `pending_translation`，仅保留仓库内容，不公开路由 |
| A4 | 统一语言切换与 hreflang 规则 | 语言切换始终指向镜像稿件，不再使用“有则跳、无则首页回退”的宽松策略 |

#### B. 路由与模板收敛

| # | 任务 | 说明 |
|---|------|------|
| B1 | 统一 URL helper | 分类/标签/文章/归档统一由 helper 生成 |
| B2 | 收敛 zh/en 页面模板重复 | article/category/tag/archive 采用共享模板或共享数据层 |
| B3 | 收敛内容查询入口 | 页面不再各自拼装过滤/排序逻辑，统一从 publish-set helper 获取 |

#### C. 运行时与资产减脂

| # | 任务 | 说明 |
|---|------|------|
| C1 | 客户端初始化按页面能力拆分 | 不再在所有页面无条件执行重脚本 |
| C2 | 搜索按语言分桶规划 | 搜索索引与 UI 要为大规模双语内容准备 |
| C3 | 字体与静态资源收口 | 清理重复字体链路、大图、低价值公共资源 |

### 验收标准

- [ ] 双语文章只有成对镜像时才生成公开详情页
- [ ] `source_id` 与 `slug` 的镜像约束在 OpenSpec 中被显式定义
- [ ] zh/en 文章语言切换不再指向缺失稿件或首页 fallback
- [ ] `src/pages` 与 `src/pages/en` 的重复路由逻辑明显减少
- [ ] 客户端初始化不再全站无差别执行所有脚本

## 与外部发布链路的关系

```text
Obsidian
   |
   v
HTTP 发布触发器
   |
   v
GitHub 仓库内容提交
   |
   v
Astro 仓库消费 Markdown 内容
   |
   +--> 双语镜像配对
   +--> 路由与 SEO 输出
   +--> 搜索索引生成
   +--> 静态构建与部署
```

关键原则：Astro 仓库不再承担“发布触发器实现”任务，而聚焦“内容消费、镜像治理、静态发布质量”。
