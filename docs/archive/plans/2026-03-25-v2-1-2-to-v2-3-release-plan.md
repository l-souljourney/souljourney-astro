# Souljourney Blog 版本拆分与执行计划（重构版）

**日期：** 2026-03-25  
**适用仓库：** `l-souljourney/souljourney-astro`  
**规划目标：** 把 Astro 仓库收敛为适合 1000+ 篇双语内容持续发布的静态出版系统，并在完成治理后再进入框架升级。

---

## 0. 基线口径（重构后）

当前统一口径如下：

1. **代码发布版本**：`2.0.0`
2. **v2.0 / v2.1**：视为已完成的基础设施与补丁阶段
3. **Astro 主线目标**：治理双语镜像发布、路由重复、搜索分桶、运行时脚本和构建性能
4. **外部发布链路**：Obsidian -> HTTP -> GitHub 已实现，不再作为 Astro 主版本目标

后续发布序列调整为：

`v2.2.0 -> v2.2.1 -> v2.2.2 -> v2.3.0`

---

## 1. 版本拆分（最终版）

### v2.2.0（P1-A：双语镜像发布治理）

**范围（必须完成）**
- 固化双语镜像规则：`source_id` 跨语言共用，`slug` 跨语言共用
- 明确公开发布集合：只有完整 zh/en 镜像对才能进入 article/RSS/sitemap/search
- 明确单边稿件策略：未配对内容只保留在仓库，不进入正式发布态
- 固化文章页语言切换与 hreflang 输出规则

**非目标**
- 不升级 Astro 主版本
- 不做 UI 风格调整
- 不处理低价值视觉重构

**验收标准**
- 公开详情页仅由完整镜像对生成
- 缺失镜像稿件不会进入 sitemap / RSS / 搜索索引
- 文章语言切换仅指向对应镜像稿件
- OpenSpec specs 已显式写入双语镜像约束

**OpenSpec change 建议名**
- `v2-2-0-bilingual-publishing-governance`

---

### v2.2.1（P1-B：路由、模板与内容查询收敛）

**范围（必须完成）**
- 提供统一 URL helper，覆盖文章/分类/标签/归档
- 收敛 `src/pages` 与 `src/pages/en` 的重复模板逻辑
- 收敛聚合查询逻辑，统一由 publish-set helper 生成列表数据
- 收敛语言路由策略，消除散落硬编码

**非目标**
- 不引入新功能模块
- 不改发布入口系统

**验收标准**
- `src/pages` 与 `src/pages/en` 的重复页面逻辑明显下降
- 链接生成不再散落在组件中手写拼接
- 分类/标签/归档输出在 zh/en 下行为一致且可预测

**OpenSpec change 建议名**
- `v2-2-1-routing-template-query-consolidation`

---

### v2.2.2（P1-C：搜索、运行时与资产减脂）

**范围（必须完成）**
- 搜索索引按语言分桶，只消费正式发布集合
- 统一客户端脚本生命周期，避免全站无差别初始化
- 清理字体、静态资源与低价值依赖冗余
- 验证大规模内容下的构建耗时与产物体积

**非目标**
- 不升级 Astro 主版本
- 不引入外部搜索服务

**验收标准**
- zh/en 搜索互不串桶
- 客户端脚本不会在页面切换中持续叠加监听
- 重点构建产物体积下降或至少完成解释性收口
- `pnpm build` / `pnpm preview` 正常

**OpenSpec change 建议名**
- `v2-2-2-search-runtime-asset-slimming`

---

### v2.3.0（Astro 6 升级窗口）

**范围（必须完成）**
- 升级运行环境与 Astro 6
- 对双语发布、路由、RSS、sitemap、搜索进行回归验证
- 视需要补最小自动化验证链路

**非目标**
- 不叠加新业务特性
- 不混入 v2.2 范围遗留重构

**验收标准**
- 升级后关键公开路径稳定
- 双语镜像模型不被升级破坏
- 关键构建与路由能力可回归验证

**OpenSpec change 建议名**
- `v2-3-0-astro6-upgrade`

---

## 2. OpenSpec 执行策略（重构后）

每个版本使用一个独立 change，遵循：

1. 每次只推进一个主 change，避免并行漂移
2. specs 先定规则，再写 proposal / design / tasks
3. task 拆分必须服务于“发布系统可维护性”，避免补丁式碎片扩张
4. 版本拆分优先按“约束层 -> 架构层 -> 性能层 -> 升级层”推进

建议节奏：

```bash
# 先完成 v2.2.0 规则层
/opsx:new v2-2-0-bilingual-publishing-governance

# 规则稳定后，再进入模板与查询收敛
/opsx:new v2-2-1-routing-template-query-consolidation

# 最后再做搜索/runtime/资产减脂
/opsx:new v2-2-2-search-runtime-asset-slimming
```

---

## 3. 发布节奏建议

- `v2.2.0` 必须先做，它定义后续所有实现边界
- `v2.2.1` 与 `v2.2.2` 可以同周期推进，但仍建议分 tag 验收
- `v2.3.0` 只做升级，不叠加治理任务

---

## 4. 风险控制与回滚

每个版本均要求：

- 独立 PR、独立验证、独立 tag
- 回滚粒度限定在单版本
- 遇到双语规则变动，优先回到 spec 层修正，而不是在页面层继续打补丁

---

v2.0 / v2.1 [已完成]

  -> v2.2.2 搜索/运行时/资产减脂
